vue2,vue3 响应式

reactive 与 ref

都是设置数据响应式(数据劫持)

1.从定义数据角度

  • ref 定义基本数据类型
  • reactive 定义对象或数组
  • ref 也可以定义对象或数组，它内部会自动通过 reactive 代理对象

2.从原理角度

  • ref 通过 Object.defineProperty()的 get()和 set()来实现响应式（数据劫持）
  • reactive 通过使用 Proxy 来实行响应式，并通过 Reflect 操作源对象内部数据

3.从使用角度

  • ref 定义数据：操作数据需要.value，读取数据时模板中直接读取，不需要.value
  • reactive 定义数据，操作数据与读取数据，均不需要.value

生命周期

  1. 最后两个钩子变动
  • beforeDestory—->beforeUnmount
  • destory——>unmounted
  1. Vue3.0 也提供了 Composition Api 形式的生命周期钩子，与 vue2.x 中的钩子对应关系如下
  • beforeCreate—->setup()
  • created—->setup()
  • beforeMount—->onBeforeMount
  • beforeUpdate—->onBeforeUpdate
  • updated—->onUpdated
  • beforeUnmount—->onBeforeUnmount
  • unmounted—->onUnmounted

批量处理

其他 Composition API

1.shallowReactive

  • 只考虑对象的第一层（浅层次）

2.shallowRef

  • 基本数据类型响应式，不进行对象的响应式处理

3.readonly

  • 设置数据为只读，不能修改

4.shallowReadonly

5.toRaw

  • 只能处理 reactive 缔造的对象，转为原始对象
  • 用于读取响应式对象对应的普通对象，对这个普通对象的所有操作，不会引起页面的变化

6.markRaw

  • 标记一个对象，使其永远不会成为响应式对象
  • 有些值比应该设置成响应式的，例如复杂的第三方库（日期，时间）
  • 不可表数据源的大列表，跳过响应式转换提高性能

7.customRef

8.provide / inject 祖与后代组件之间的通信

9.响应式数据的判断

  • isRef
  • isReactive
  • isReadonly
  • isPorxy

Composition API 的优势

1.options API 存在的问题

使用传统 OptionAPI 中的，新增或者修改一个需求没救需要分别在 data,methods,computed 中修改

2.Composition API 的优势

可以更加优雅的组织我们的代码，函数，让相关功能的代码更加有序的组织在一起

新组件

1.Fragment（Vue3 中，组件可以没根标签）
2.Teleport

3.defineAsyncComponent(异步引入组件)

4.Suspense
